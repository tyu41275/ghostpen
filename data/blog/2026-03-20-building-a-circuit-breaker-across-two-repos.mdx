## Introduction

I've been building an AI agent called AutoAgent that watches gaming VODs and extracts knowledge from them using vision APIs. The whole pipeline runs through a library I wrote called llm-router, which handles multi-provider LLM routing — Anthropic, OpenAI, and Google. For the past few months, Anthropic has been my primary provider because they're the only ones that support batch processing, which cuts vision API costs in half.

The problem is, when Anthropic goes down (and it does — 500s, 529s, the occasional 503), everything stops. The Watch phase can't extract frames, the knowledge base doesn't grow, and the whole learning loop grinds to a halt. I wanted to build graceful failover to OpenAI and Google so that when Anthropic is having a bad day, work continues on another provider.

## The Problem

The failover logic needed to live in the right place. My first instinct was to handle it in AutoAgent's WatchLoop — just catch the exception and retry with a different provider. But that would mean encoding provider-specific knowledge (HTTP status codes, retry semantics) above the abstraction boundary. AutoAgent submits `LLMRequest` objects and gets `LLMResponse` objects. It shouldn't need to know whether Anthropic is returning 529s.

So the circuit breaker went into llm-router, where it belongs. The router already selects providers based on rate limit headroom. It already tracks errors per provider. It just didn't _use_ that error signal to avoid sick providers.

## The Plan That Got Replanned

I originally had 11 issues across the two repos. The problem was they were too broad — titles like "Circuit breaker state + error classification on \_ProviderState" bundled multiple concerns into one task. No explicit file paths, no verification steps, no acceptance criteria. When I tried to build from these specs, I spent more time figuring out what the task actually wanted than writing code.

So I scrapped them and replanned. The new plan had 9 issues — 4 in llm-router (#16-#19) and 5 in AutoAgent (#55-#59). Each issue was atomic: max 3 files, explicit file paths with reasons, verification commands, and acceptance criteria checkboxes. Two fewer issues, but the boundaries were clean.

## Building the Circuit Breaker

The implementation started with error classification (#16). Before the router could make decisions about provider health, it needed to understand _what kind_ of error it was looking at. A 500 is a server error — the provider is sick. A 429 is a rate limit — already handled by headroom tracking. A 400 is a client error — retrying won't help. I wrote 30 tests for this, TDD style.

The review caught four issues in that first task alone. A type guard crash when the error wasn't an `APIStatusError`. A `DeadlineExceeded` being misclassified as a server error when it's actually a timeout. Dead code from an earlier iteration. And an ambiguity with HTTP 408 that could be either client or server depending on context.

The circuit breaker itself (#17) was the meatiest task — another 30 tests. After 3 consecutive failures, the provider gets marked unhealthy for a cooldown period. The review found four critical issues: a thundering herd problem where multiple concurrent requests could all trigger recovery probes simultaneously, a cooldown timer that reset on every failure instead of staying fixed, an interleaved error ambiguity where successes from other providers could reset the counter prematurely, and an unbounded error counter that could theoretically overflow.

All caught by the review step. All fixed before shipping.

The half-open probe (#18) was mostly filling gaps from the circuit breaker — adding the `_is_half_open()` method, a `RECOVERY_COOLDOWN` constant, renaming `probe_in_flight` to `half_open_lock` for clarity. Clean pass, no fix loops.

Failover metadata (#19) added `was_failover` and `failover_reason` to `LLMResponse` so the consuming application knows when a response came from a backup provider, and a `provider_health()` method on `LLMRouter` for health monitoring. Clean pass. 84 tests total across the library at this point.

## Wiring It Into AutoAgent

With the llm-router phase done, the AutoAgent side was mostly plumbing. Database migration (#55) added `provider_used` and `was_failover` columns to the jobs table. The WatchLoop wiring (#56) aggregated provider attribution across a batch of frames and made sure `create_router()` always includes fallback providers.

The UI work was three tasks: a health endpoint (#57), provider badges on job cards (#58), and a health indicator in the header (#59) with colored dots per provider and an expandable dropdown. All clean passes — no fix loops on any of the AutoAgent tasks.

## What I Learned

The whole feature shipped across 9 issues, 8 PRs, and two repos. The llm-router side had 84 tests. The review step caught 8 critical issues across the first two tasks — issues that would have been bugs in production. The fix loop pattern (review finds issue → build fixes → re-review) resolved every one of them before they reached QA.

The replanning was worth it. The first set of 11 issues would have taken longer to build from because I would have been constantly figuring out task boundaries mid-implementation. The second set of 9 issues with explicit file paths and verification steps meant I could focus on writing code instead of interpreting specs.

## Conclusion

The circuit breaker is live. When Anthropic goes down, the router switches to OpenAI or Google automatically. When it comes back, a half-open probe verifies recovery before routing traffic back. The UI shows which provider served each job and whether failover was involved. And the whole thing is transparent — users know exactly what's happening under the hood.
