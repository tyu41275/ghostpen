## Introduction

I have been building an AI agent called AutoAgent that watches gaming VODs and extracts knowledge from them using vision APIs. The whole pipeline runs through a library I wrote called [llm-router](https://github.com/tyu41275/llm-router), which handles multi-provider LLM routing for Anthropic, OpenAI, and Google. Anthropic has been my primary vision provider because they are the only ones that support batch processing, which cuts costs in half.

The problem? When Anthropic goes down with 500s, 529s, or the occasional 503, everything stops. The Watch phase cannot extract frames, the knowledge base does not grow, and the whole learning loop grinds to a halt.

I wanted graceful failover so that when Anthropic is having a bad day, work continues on another provider. Sounds straightforward. It was not.

## The Problem

My first instinct was to handle failover in AutoAgent's WatchLoop -- just catch the exception and retry with a different provider. Something like:

```python
try:
    response = await router.complete(request)
except AnthropicError:
    request.provider = "openai"
    response = await router.complete(request)
```

But that would mean encoding provider-specific knowledge -- HTTP status codes, retry semantics -- above the abstraction boundary. AutoAgent submits `LLMRequest` objects and gets `LLMResponse` objects back. It should not need to know whether Anthropic is returning 529s or if OpenAI rate limit headers look different from Google's.

So the circuit breaker had to go into llm-router. The router already selects providers based on rate limit headroom via `_select_provider()`. It already tracks errors per provider. It just did not _use_ that error signal to avoid sick providers.

Ok. So where exactly does the circuit breaker state live?

## Error Classification -- The Foundation

Before the router can decide a provider is unhealthy, it needs to understand _what kind_ of error it is looking at. I started with a classification function:

```python
def _classify_error(exc):
    # Timeouts are their own category -- not a server failure
    timeout_types = ("TimeoutError", "APITimeoutError", "DeadlineExceeded")
    if type(exc).__name__ in timeout_types:
        return "timeout", None

    status_code = getattr(exc, "status_code", None)
    if status_code in (500, 502, 503, 529):
        return "server_error", status_code
    if status_code == 429:
        return "rate_limit", status_code    # Already handled by headroom
    if 400 <= status_code < 500:
        return "client_error", status_code  # Retrying won't help
    return "unknown", status_code
```

A 500 is a server error -- the provider is sick. A 429 is a rate limit -- already handled. A 400 is a client error -- retrying will not help, so it should not count toward the circuit breaker. I wrote 30 tests for this, TDD style, red-green.

The review immediately caught something I missed. I had `DeadlineExceeded` classified as a server error. But it is actually a timeout -- the request did not fail on the server side, it just took too long. Different classification, different circuit breaker behavior. Small distinction, big difference if you are making health decisions based on it.

## The Circuit Breaker

The breaker state lives on `_ProviderState` -- the per-provider tracking object that already existed for rate limit headroom:

```python
class _ProviderState:
    BREAKER_THRESHOLD = 3
    COOLDOWN_SECONDS = 300  # 5 minutes, fixed

    def __init__(self, provider):
        self.provider = provider
        self.consecutive_errors = 0
        self.unhealthy_since = None
        self.half_open_lock = False

    def _is_half_open(self):
        if self.unhealthy_since is None:
            return False
        return (time.time() - self.unhealthy_since) >= self.COOLDOWN_SECONDS
```

Three consecutive failures and the provider gets marked unhealthy. After a 5-minute cooldown, it enters half-open state -- one probe request gets through to test if the provider has recovered.

Sounds clean, right? The review found four critical issues in this code alone.

## The Bugs the Review Caught

**Thundering herd on the probe.** When a provider's cooldown expires, `_select_provider()` checks `_is_half_open()` and sends a probe request. But in a concurrent system, multiple requests could all see half-open as true simultaneously and all send probes at once -- hammering a recovering provider with a burst of traffic. Ok. That is not good.

The fix was a lock boolean:

```python
for state in self._states:
    if state._is_half_open() and not state.half_open_lock:
        state.half_open_lock = True   # Only one probe at a time
        return state, False, None
```

This works because `_select_provider()` is synchronous -- no `await` points, so the check-then-set is atomic under asyncio. I almost reached for `asyncio.Lock` before realizing that was overkill for a synchronous code path.

**Cooldown reset bug.** Every time a new error came in while the provider was already unhealthy, I was resetting `unhealthy_since` to the current timestamp. This meant the 5-minute cooldown kept restarting, and a provider under sustained load would _never_ get a recovery probe. The fix: only set `unhealthy_since` on the transition from healthy to unhealthy, not on subsequent errors.

**Interleaved error ambiguity.** If provider A fails and then provider B succeeds, should B's success reset A's consecutive error count? Obviously not -- each provider's counter is independent. But I had a code path where a successful response on _any_ provider was clearing `consecutive_errors` globally. That would have been a subtle, hard-to-reproduce bug in production.

**Unbounded counter.** `consecutive_errors` just kept incrementing. Not a bug in practice since we only check `>= BREAKER_THRESHOLD`, but the reviewer flagged it as a correctness issue. I capped it at the threshold value. Fair enough.

## Wiring It Into AutoAgent

With 84 tests passing in llm-router, the AutoAgent side was mostly plumbing. Database migration added `provider_used` and `was_failover` columns to the jobs table:

```sql
ALTER TABLE jobs ADD COLUMN provider_used TEXT;
ALTER TABLE jobs ADD COLUMN was_failover BOOLEAN DEFAULT FALSE;
```

The WatchLoop now aggregates which providers served which frames in a batch. And `create_router()` always includes fallback providers -- before this change, it only configured the one you selected in the UI.

The UI work was three tasks: a health endpoint (`GET /api/provider-health`), provider badges on job cards showing which provider served the response, and colored dots in the header showing each provider's circuit breaker state. All clean passes -- no fix loops on any of the frontend tasks. Cool!

## What I Learned

The review step caught 8 critical issues across the first two tasks. The thundering herd bug in particular is the kind of thing you would only discover under load, and by then it is causing the exact cascading failure the circuit breaker was supposed to prevent.

The replanning was also worth calling out. My first task breakdown produced 11 issues with vague descriptions -- titles like "Circuit breaker state + error classification on \_ProviderState" that bundled multiple concerns. No file paths, no verification steps. The second attempt produced 9 issues with explicit file paths, verification commands, and acceptance criteria. Fewer issues, but each one was actually completable without guessing at scope.

## Conclusion

The circuit breaker is live. When Anthropic goes down, the router switches providers automatically. When it recovers, a half-open probe verifies before routing traffic back. And the whole thing is transparent -- you can see which provider served each job and whether failover was involved.
